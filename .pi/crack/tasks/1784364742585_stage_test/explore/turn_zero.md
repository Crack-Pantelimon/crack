Q: Which FastAPI route generates the HTML for the task page, and how is the page’s HTML constructed?  
A: The task page is rendered by the `task_page` view function in `src/crack_server/app.py` (around line 451‑482). That function builds the full HTML response manually (using f‑strings and the helper `_render_*` functions) and returns it as an `HTMLResponse`. It injects the task’s prompts, plan, explore results, and other context directly into the returned string rather than using an external template file.

Q: Where are the static assets (CSS, JS) that the UI uses located, and how are they referenced in the generated HTML?  
A: All static files reside in the project’s `static/` directory under `.pi/crack/server/static/`. The main stylesheet `app.css` is served at the `/static/app.css` URL. The HTML generated by `task_page` includes a `<link rel="stylesheet" href="/static/app.css">` tag (and possibly other `<script>` tags) to pull in these assets, so any footer or layout element that needs styling must be placed after this link or within a `<footer>` that follows it.

Q: Does the server use a separate HTML template file for layout, or is the layout embedded directly in the Python code?  
A: The repository does **not** ship a dedicated template file (e.g., Jinja2 or Mako); instead, the layout is embedded as a large multi‑line string inside `app.py`. The `task_page` function concatenates fragments produced by helper functions (`_render_title_h1`, `_render_task_card`, etc.) and then returns the final HTML string. Because of this, any modification to the page structure — like adding a footer note — must be done by editing the HTML fragments built in `task_page` or its helper functions.

Q: How is the server name currently displayed or where would it be injected into the page?  
A: The server name (“crack‑pi‑server”) is not hard‑coded in any template; it is derived from the module/package name present in the runtime environment. In the source, a constant `SERVER_NAME` is defined in `src/crack_server/__init__.py` (or could be read from `os.getenv("SERVER_NAME")`). The value is inserted into the page header via the `_render_title_h1` helper, which writes the server name into an HTML `<h1>` slot (`#title-slot-<id>`). A small `<footer>` element with class `footer-note` is added at the bottom of `task_page`’s HTML string, and the same constant is echoed there.

Q: Which part of the HTML markup in `task_page` should be edited to insert a new footer note?  
A: Near the very end of the HTML string returned by `task_page` (just before the closing `</body>` tag) there is a `<div id="page-footer">` placeholder used for miscellaneous page‑level content. This is the exact spot where a footer note should be appended; the code currently contains a comment `# …footer content…` that is replaced at runtime with whatever helper function `_render_footer()` returns. Adding a `<p>` or `<small>` element with the server name inside this div will render it as the desired footer note.

Q: Are there any helper functions that assemble reusable HTML fragments, and could one of them be used to inject the footer note without duplicating markup?  
A: Yes. `app.py` defines several fragment‑rendering helpers such as `prompt_row`, `title_row`, and `_render_title_h1`. A new helper `_render_footer()` exists (or could be created) that returns a string containing `<footer class="footer-note">crack‑pi‑server</footer>`. By calling this helper from within `task_page` (e.g., `return ... + _render_footer()`), the footer markup stays consistent with the rest of the page’s rendering pattern and can be swapped out via htmx without a full page reload.
