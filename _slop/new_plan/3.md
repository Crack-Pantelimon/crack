## 3.1 Driver mesh inside the car

Today `tick_entering_car` despawns the pedestrian entirely at timer end (interaction_ui.rs:174–183), so the car drives empty. bugs2.txt: the mesh should be parented to the car and play a driving loop.

**Fix:**
- New component `DriverMesh { car: Entity }` and event handling inside the pedestrian plugin (this is the "best place": the mesh, animations, and GLB URL all live there; the car side only exposes `ActivePlayerVehicle`).
- When `EnteringCarTimer` finishes: instead of despawning everything, despawn the pedestrian's *physics* (controller, `RigidBody`, `Collider`, character-controller components) and re-use its already-loaded visual scene: strip it from the controller entity, insert `DriverMesh`, `ChildOf(car_entity)` (car is spawned via `WorldAssetRoot`, but a plain child with `Transform` works — the previous collider problem came from `ColliderConstructorHierarchy` walking *mesh* children; verify no collider is generated for the driver mesh child, and if it is, insert `ColliderConstructorHierarchy`-opt-out / spawn the mesh as a fresh GLB scene child marked to be skipped).
- Local transform = `CarSeatOffset` resource (see 3.2). Play the driving-loop clip; the animation clip list in animation.rs:186–191 already loads by name — add the loop clip name (inspect the GLB with the same python-gltf snippet used in the chat log if the exact name is unknown; expected something like `Sitting_Idle`).
- On exit: despawn the `DriverMesh` child after the exit sequence starts (3.4).

## 3.2 In-car Debug seat-offset menu

Planned in car_animations.md (`CarSeatOffset` resource + egui window) — never written; `grep CarSeatOffset` finds nothing.

**Fix:**
- `#[derive(Resource)] CarSeatOffset { pub offset: Vec3, pub y_rot: f32 }` with defaults tuned once in-game (start `(-0.35, 0.15, 0.0)`, `y_rot = PI` per the chat's 180° finding).
- Egui window "Debug → Car Seat", three translation sliders (−2..2) plus the Y-rotation slider; registered in `EguiPrimaryContextPass` with `run_if(in_state(GameControlState::DrivingCar))` — same pattern as `speedometer_ui` registration (mod.rs:49–52). Only visible while driving, per bugs2.txt.
- A `Last`-schedule-adjacent system copies the resource into the `DriverMesh` child's local `Transform` when changed.

## 3.3 Entry check: crosshair + 1 m (bugs2.txt) instead of 3 m proximity sphere

`detect_car_interaction` (interaction_ui.rs:119–130) takes the nearest car within **3 m**, no look direction.

**Fix:** raycast from the player camera along its forward (`SpatialQuery::cast_ray`, filter to the `Car` physics layer — the layer already exists: `GamePhysicsLayer::Car`), max distance ~2.5 m from camera (≈1 m from the character), and require the hit entity's root to be a `Car`. Map collider child → car root via `ChildOf` walk (helper likely exists in weapons hit-detection code — reuse it; the weapons plugin resolves hit entities to roots the same way). Keep the existing `Without<EnteringCarTimer>` guard.

## 3.4 Exit sequence symmetry

Today `handle_exit_car` (interaction_ui.rs:208–233) instantly spawns the pedestrian 2 m to the right; `Sitting_Exit` plays only via the `is_exiting_car` flag on the freshly spawned controller, with no movement out of the car.

**Fix (mirror of enter):**
1. F while driving → remove `ActivePlayerVehicle` (car keeps physics and coasts — Chapter 1 root cause 6 makes this true), keep the `DriverMesh` child but detach it to world space at its current pose.
2. Give it an `ExitingCarTimer`; over 1.2 s interpolate translation from seat → door → 1.5 m beside the car while `Sitting_Exit` plays (reuse the lerp pattern of `tick_entering_car` interaction_ui.rs:150–171, reversed).
3. On timer end: despawn the mesh, trigger `SpawnControlledPedestrianEvent` at the final pose (existing call, interaction_ui.rs:222–228), switch state to `ControllingPedestrian`.
4. All entity despawns/inserts through `commands.get_entity(..)` guards — the despawn-warning from the chat log (`Entity despawned: … 1911v17`) comes from unguarded commands on dead entities; audit `tick_entering_car`/`tick_exiting_car`/`handle_exit_car` for remaining unguarded `commands.entity(..)` calls.

## 3.6 Gizmo correctness (`draw_car_gizmos`, mod.rs:854–927)

Thresholds are hardcoded `1.0` m (mod.rs:871, 879, 896) while the actual max ray length is a slider (default 1.15) — rays between 1.0 and 1.15 m currently render as misses. Use `drive_state.max_ray_length` (the query already has `CarDriveState`; the `_drive_state` binding at mod.rs:858 is sitting there unused). Draw ray lines to `max_ray_length`, engaged markers per the same validity rule the physics uses (post-median-filter). Keep gizmos registered state-independently so `car_sim` shows them (already true, mod.rs:26–38).
