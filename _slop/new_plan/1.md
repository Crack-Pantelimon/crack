# Chapter 1 — Driving physics: stop the violent seam launches

## Context

The map is made of tiles whose edges have near-vertical *skirts* and small gaps between neighboring tiles. The car is a single dynamic rigid body levitated by 4 "virtual wheels", each sampling the ground with a 3×3 grid of downward rays (9 rays per wheel, 36 total). This model is intentional and stays — **no physics springs/wheel colliders**. The problem: at high speed, crossing a tile seam launches the car violently upward.

All code references are to
`crack_demo/demo_resolution_selector_web_bevy/src/plugins/cars_driving/driving_plugin/mod.rs`
at commit `394e655` unless stated otherwise.

The seam launch is not one bug — it is a chain of four formula problems that amplify a single bad frame into a 20 G impulse. Fix them all; each one individually is enough to cause visible bumps.

## Root cause 1 (worst): depenetration teleport — `apply_car_steering_and_drive`, mod.rs:615–623

```rust
if max_deficit > 0.0 {
    let push_dir = (avg_normal / engaged_wheels_count as f32).normalize_or_zero();
    if push_dir.length_squared() > 0.001f32 {
        car_transform.translation += push_dir * max_deficit;
    }
}
```

`max_deficit` is computed (mod.rs:511–518) whenever any wheel's *average* engaged ray length dips below `(suspension_min + suspension_rest) / 2`. With the defaults (`max_ray_length = 1.15`, `rest = 50% = 0.575`, `min = 0.3·rest = 0.1725`) that threshold is **0.374 m** — easily crossed by one frame of bad ray data at a seam. The fix teleports the rigid body's `Transform` directly:

- position changes but **velocity does not** → the physics solver sees an unexplained displacement;
- the next frame's rays are cast from the new position → lengths read long → springs push differently → oscillation;
- at 100+ km/h a front wheel crossing a seam step triggers this every time the tiles disagree by a few cm.

**Fix:** delete the teleport block and the `max_deficit` bookkeeping entirely (mod.rs:480, 511–518, 615–623). The progressive spring (root cause 3, fixed form) is responsible for keeping the body out of the ground. Only if end-to-end verification shows the chassis tunneling into the map, add a *velocity-space* correction instead — e.g. cancel downward velocity along the contact normal, clamped to a small budget (`≤ 2 m/s per frame`), never a position write.

## Root cause 2: min-anchored outlier filter — `update_wheel_contact_normals`, mod.rs:830–842

```rust
let max_valid_dist = min_hit_dist + 0.25;
for i in 0..9 {
    if distances[i] <= max_valid_dist { ... keep ... }
    else { distances[i] = f32::MAX; hit_points[i] = Vec3::ZERO; }
}
```

Intent was "ignore rays that fell through a crack" (long hits). But anchoring the window at the **minimum** backfires at seams: when the next tile is a few cm *higher*, the shortest hit lands on the higher tile, and every legitimate hit on the tile the car is actually on is > `min + 0.25` away → discarded. The wheel's average length collapses to the single short hit in one frame → spring spike + (currently) the teleport of root cause 1.

**Fix:** anchor the validity window at the **median** hit distance:

1. Collect the `k` actual hits (`k = raw_hits_count`), sort their distances, take `median = sorted[k/2]`.
2. Keep rays with `|d - median| <= 0.25`; mark the rest as misses (`f32::MAX`).

The median is robust to both failure modes: a lone short hit on a skirt lip *and* a lone long hit through a crack get discarded; the majority surface wins. Keep the existing "no hit at all → `f32::MAX`" behavior.

## Root cause 3: unbounded hyperbolic spring scaling — mod.rs:529–535

```rust
let scaling = if avg_length < drive_state.suspension_rest {
    let remaining = (avg_length - drive_state.suspension_min).max(0.001f32);
    ((drive_state.suspension_rest - drive_state.suspension_min) / remaining).max(1.0f32)
} else { 1.0f32 };
```

As `avg_length → suspension_min`, `remaining → 0.001` and `scaling → (0.575−0.1725)/0.001 ≈ ×400`. This multiplier is applied to **both** the spring force (mod.rs:536) and the damping force (mod.rs:544), capped only at 5 G per wheel (mod.rs:548) — i.e. up to **20 G total** on a 1200 kg car. One frame of short ray data = launch.

**Fix:**
- Replace the hyperbola with a bounded progressive curve on compression ratio `c = ((rest − length) / rest).clamp(0, 1)`:
  `scaling = 1.0 + 2.0 * c * c` (max ×3 at full compression). Expose the `2.0` as the `Stiffness scale` slider (see Chapter 3).
- Apply `scaling` to the **spring term only**. Damping stays at its critically-damped value `2·√(k·m)·damping_ratio` — scaling damping is what turns a bump into a hammer blow, because damping force is proportional to impact velocity which is already large at seams.
- Lower the per-wheel clamp from 5 G to **2.5 G** (mod.rs:548): `max_susp_force = mass_per_wheel * 9.81 * 2.5`. 10 G total is still plenty to absorb a curb; 20 G is a catapult.
- Keep `total_suspension_force.max(0.0)` — suspension must never pull the car down.

## Root cause 4: per-wheel averaging is discontinuous → switch to per-ray springs

Currently each wheel averages its engaged ray distances (mod.rs:492–506) and drives one spring from that average. When a ray flips between engaged/discarded (seam gap, skirt, filter), the average jumps discontinuously even with the median filter.

**Fix — per-ray force accumulation** (this is still "distance + surface contact area", exactly the model bugs2.txt mandates; no physics wheels are introduced):

For each wheel, for each of its 9 rays that has a valid hit:

```
ray_share      = mass_per_wheel / 9.0                       // kg supported by this ray at full contact
displacement   = (suspension_rest + extra_spring_length) - ray_distance
base_stiffness = (ray_share * 9.81) / extra_spring_length   // balances gravity at rest, per ray
c              = ((suspension_rest - ray_distance) / suspension_rest).clamp(0.0, 1.0)
spring         = base_stiffness * displacement * (1.0 + stiffness_scale * c * c)
damping        = -2.0 * sqrt(base_stiffness * ray_share) * damping_ratio * vel_at_ray.dot(Vec3::Y)
force          = Vec3::Y * (spring + damping).max(0.0).min(ray_share * 9.81 * 2.5)
```

- `vel_at_ray` = body velocity at the ray origin (`lin_vel + ang_vel × (ray_origin_world − com_world)`), reusing the existing pattern from mod.rs:487–488.
- Apply each ray's force at its **ray origin** (not the hit point — the origin is rigidly attached to the body, which keeps torque arms stable): `total_torque += (ray_origin - com_world).cross(force)`.
- Rays with no valid hit contribute **zero**. A wheel half-over a tile gap thus automatically produces ~half force — smooth, proportional to real contact area, no cliff.
- Contact normal stays hardcoded `Vec3::Y` (mod.rs:849) — correct decision already made; tile skirt normals are garbage. Keep using per-ray hit *positions* only.
- The per-wheel notions used elsewhere remain: a wheel is "grounded" if ≥1 of its rays has a valid hit (traction gate at mod.rs:395–421 keeps requiring ≥2 grounded wheels for drive/brake/lateral forces — keep that logic unchanged).
- `avg_suspension_height` (mod.rs:616) becomes the mean of valid ray distances across all wheels (used by UI/wheel placement).

## Root cause 5: no temporal smoothing of suspension length

bugs2.txt: input integration was moved to 0.060 s and "other things that integrate/average" should have a "relatedly short" period. Ray data currently has none — one frame of noise passes straight into forces.

**Fix:** low-pass the per-ray spring input distance with a ~0.06 s time constant:
`smoothed = smoothed + (raw - smoothed) * (dt / 0.06).min(1.0)`, stored in `WheelContactData` (add `smoothed_distances: [f32; 9]`). Rules:
- When a ray transitions miss→hit, reset its smoothed value to the raw hit (don't lerp from `f32::MAX`).
- When a ray misses, it contributes no force (do not smooth "towards infinity"; just freeze/flag it).
- Gizmos should keep drawing **raw** hits so seam artifacts stay visible while tuning.

## Root cause 6: stale input — driverless cars keep accelerating

`avg_accelerate/avg_brake/avg_steer` are only recomputed inside `car_drive_observer` (mod.rs:222–282), which runs only when a `Drive` event arrives — and `keybinds_control_car` only triggers events for the `ActivePlayerVehicle` (keybinds_control.rs:101–106). Exit the car while holding W → the averages stay frozen non-zero forever → the empty car drives away (into the physics of Chapter 3's exit sequence, this reads as "cars behave insanely after exit").

**Fix:** at the top of `apply_car_steering_and_drive`, per car, expire the history by timestamp using the same 0.060 s window, and when the history is empty set `avg_accelerate = avg_brake = avg_steer = 0.0` and decay `current_steer_integrated` toward 0 with the existing `shrink_rate = 5.0`. (Either factor the window logic into a helper shared with `car_drive_observer`, or move the whole averaging into `apply_car_steering_and_drive` and let the observer only push raw events into `history`. The second option is cleaner — one integration site.)

## Runtime suspicion to verify (not a code change yet): chassis collider vs. seam steps

The car body has a `ConvexDecompositionFromMesh` collider colliding with `Map`, plus `SweptCcd` (spawn_car.rs:94–104). At rest the body bottom clears the ground by roughly `suspension_rest − 0.10·car_height` ≈ 0.4 m, so it *shouldn't* touch — but under heavy compression or on a big step the hull can strike the vertical seam wall directly, and CCD makes that contact resolve aggressively at speed.

**Verification step:** in `car_sim`, log/visualize collision events involving the car body (avian `CollisionStarted` events) while driving seams fast. If body↔map contacts fire during normal driving, the fallback is to raise `suspension_rest` default or shrink the hull's bottom (not to remove body↔map collision — it's needed for crashes).

## Slider changes (implemented here, listed in Chapter 3's UI audit too)

- Keep: `Max Ray Length` (0.60–1.80 m, default 1.15), `Rest Length %` (10–90, default 50), `Max Speed`, `Horsepower`.
- Add: `Stiffness scale` (0.0–6.0, default 2.0 — the `c²` coefficient), `Damping ratio` (0.2–2.0, default 0.8 — replaces the hidden `suspension_damping`).
- Derivations that stay per-frame in `apply_car_steering_and_drive` (mod.rs:330–332): `suspension_rest = max_ray_length · rest%`, `traction_loss_threshold = max_ray_length`. `suspension_min` disappears with the hyperbola (the bounded curve doesn't need it).

## Verification checklist

1. `cargo check` then run the `car_sim` binary (`src/bin/car_sim.rs`); gizmos must show the 3×3 ray grids (orange), hits (blue), contact patches (green).
2. Drive along a road crossing multiple tile seams at > 100 km/h. Pass: no vertical launches; at most a small ripple. Watch the gizmo patches stay green across seams.
3. Park half the car over a tile gap: car should settle tilted slightly, not oscillate.
4. Drive off a ledge: with < 2 wheels grounded, no drive/steer force (existing gate), clean ballistic flight, soft landing without the 5 G+ pogo.
5. Exit the car (F) while holding W: the car must coast to a stop from drag/rolling resistance, not drive away.
6. Sanity: RPM/gear numbers in the UI still move plausibly (engine sim at mod.rs:348–388 is untouched by this chapter).
