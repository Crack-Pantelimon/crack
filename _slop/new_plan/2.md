# Chapter 2 — Cosmetic wheels done right

## Context

Wheels are purely cosmetic: standalone entities (GLB scenes `car-wheel_00003_` / `car-wheel_00005_`, picked per car at spawn, scaled 0.6) with **no physics**, positioned every frame from the car's ray-contact data. That architecture stays. Two visible defects remain: the wheels **spin around the car's forward axis** (screenshot confirmed), and their update timing/radius handling is sloppy.

Code references:
- `crack_demo/demo_resolution_selector_web_bevy/src/plugins/cars_driving/driving_plugin/mod.rs` — `CosmeticWheel`, `update_cosmetic_wheels` (mod.rs:662–746)
- `.../driving_plugin/spawn_car.rs` — wheel spawning (spawn_car.rs:124–139)

## Bug 1: spin axis — the quaternion composition is in the wrong order

Current code (mod.rs:732–743):

```rust
let base_y_rot = if is_right { FRAC_PI_2 } else { -FRAC_PI_2 };
let total_y_rot = base_y_rot - steer_angle;
let rot_y = Quat::from_rotation_y(total_y_rot);
let rot_x = Quat::from_rotation_x(cosmetic_wheel.accumulated_rotation);
wheel_transform.rotation = car_transform.rotation * rot_y * rot_x;
```

Why this rotates around the forward axis: quaternion multiplication `rot_y * rot_x` applies `rot_x` **first**, in the wheel model's own frame, and then yaws the whole thing by ±90°. A +90° yaw maps the model's X axis onto −Z; −90° maps it onto +Z. So the "spin about X" ends up spinning about the **car's forward axis** — exactly the tumbling in the screenshot. The spin *rate* looks right because only the axis is wrong.

**Fix — put the spin between the car frame and the base yaw:**

```rust
let base_y_rot = if is_right { FRAC_PI_2 } else { -FRAC_PI_2 };
wheel_transform.rotation = car_transform.rotation
    * Quat::from_rotation_y(-steer_angle)                       // steering (front wheels only, else 0)
    * Quat::from_rotation_x(cosmetic_wheel.accumulated_rotation) // roll about car-local X (the axle)
    * Quat::from_rotation_y(base_y_rot);                         // orient model cylinder axis onto the axle
```

Reading right-to-left: first orient the GLB (its cylinder axis, +Y in Blender → ±Z in Bevy, turned onto the car-local X axle), then roll about the axle, then steer, then car orientation. Because the roll now happens about car-local X *after* the base yaw, **one sign of `accumulated_rotation` is correct for both left and right wheels** — do not negate per side. Steering sign (`-steer_angle`) matches the previous `base_y_rot - steer_angle` behavior, which the chat confirmed as "spin and steer is correct" for direction.

If after this the wheels roll *backwards* when driving forward, flip the sign in one place only: `accumulated_rotation -= rot_speed * dt` (do not touch the composition).

## Bug 2: two different wheel radii in the same function

- Spin rate uses `wheel_radius * 0.6` (mod.rs:720) — accounting for the 0.6 spawn scale.
- Ground offset uses full `wheel_radius` (mod.rs:707–713) — so the wheel hub floats `0.4·r` too high whenever grounded.

**Fix:** measure the real rendered radius once and use it everywhere.
- Add a `measured_radius: Option<f32>` to `CosmeticWheel`. A small system (same deferred pattern as `init_cars_system` in spawn_car.rs:150–220, keyed on a `NeedWheelBoundsCompute` marker) walks the wheel entity's loaded meshes, computes the max extent perpendicular to the cylinder axis in local space, multiplies by the 0.6 scale, and stores it.
- Until measured, fall back to `drive_state.wheel_radius * 0.6` for both spin and offset — the same value in both places.
- Delete the divergent uses of raw `wheel_radius` in this function.

## Bug 3: update runs too early in the frame

`update_cosmetic_wheels` is chained in `Update` (mod.rs:26–38) and reads the car's `Transform`. Avian writes physics results and Bevy propagates transforms later, so wheels lag the chassis by a frame at speed. bugs2.txt explicitly requires: *"position will be computed from the car data in the **Last** schedule to make sure we render them in the correct place at all times."*

**Fix:**
- Move `update_cosmetic_wheels` registration from the `Update` chain to `app.add_systems(Last, update_cosmetic_wheels)` (it stays state-independent, as now).
- Read the car's `GlobalTransform` instead of `Transform` (car is a root entity so values match, but `GlobalTransform` is the render-facing source of truth and is what avian's interpolation writes for rendering).
- Since the wheel entities are roots, writing their `Transform` in `Last` still reaches the renderer for the same frame after transform propagation… it does **not** — propagation has already run in `PostUpdate`. Therefore write the wheel's `GlobalTransform` directly as well (`*wheel_gt = GlobalTransform::from(computed_transform)`), or schedule the system in `PostUpdate` ordered `after(PhysicsSet::Sync)` and `before(TransformSystem::TransformPropagate)`. **Prefer the `PostUpdate` ordering variant** — it is the idiomatic Bevy way and avoids hand-writing `GlobalTransform`; it satisfies the intent of "Last" (after physics, before render extraction).

## Bug 4: orphaned wheels never die

`update_cosmetic_wheels` silently skips wheels whose `parent_car` lookup fails (mod.rs:679–681). If a car despawns, its 4 wheels freeze in the air forever.

**Fix:** in that failure branch, despawn the wheel entity (`commands.entity(e).despawn()` — guarded with `commands.get_entity` per the despawn-warning lesson from the chat log).

## Position formula (keep, with one clamp)

The grounded case is already what we want (mod.rs:698–716): corner XZ offsets at `±(car_half_width + 0.1)`, `±car_half_length`; Y from `ray_start_y_offset − avg_engaged_ray_distance + radius`, transformed by the car transform. Changes:
- Use Chapter 1's **smoothed, median-filtered** per-wheel distances (the same data the springs use) so wheels don't jitter at seams.
- Airborne case currently drops the wheel by the full `traction_loss_threshold`; clamp droop to `suspension_rest + 0.1` so wheels hang naturally instead of reaching for the ground.
- Use the measured radius from Bug 2.

## Verification checklist

1. `car_sim`: drive forward — wheels roll about their axles (tread moves in the driving direction), no tumbling around the forward axis. Reverse: they roll backwards.
2. Steer at low speed: front wheels yaw with the steering, rear wheels don't; rolling continues about the steered axle.
3. High speed over seams: wheels stay glued to the ground contour, hubs at ground + measured radius (no floating, no clipping), no one-frame trailing behind the chassis.
4. Drive off a ledge: wheels droop slightly and hold; they do not stretch to the ground.
5. Despawn/respawn the car from the menu: old wheels disappear with it.
